# 결 Ledger · 기획서 초안 (v0.1)

> 개인과 사업의 돈 흐름을 한 곳에서. 1인 사업가·프리랜서·소상공인을 위한 가벼운 자산관리 ERP.

---

## 1. 한 줄 정의 (Positioning)

가계부보다는 똑똑하고, 대기업 ERP보다는 가벼운 **"하이브리드 머니 매니저"**.
사업과 개인의 돈을 **분리하되 한 화면에서** 본다는 점이 핵심.

| 구분 | 일반 가계부 | 대기업 ERP | **결 Ledger** |
|------|------------|-----------|--------------|
| 무게감 | 가벼움 | 매우 무거움 | 가벼움 |
| 사업 손익 관리 | ✕ | ◎ | ○ (간이 손익) |
| 개인/사업 분리 | ✕ | △ | ◎ |
| 세금 신고 참고 | ✕ | ◎ | ○ (카테고리 분류·CSV) |
| 진입 장벽 | 낮음 | 높음 | 낮음 |

## 2. 핵심 기능 리스트 & 우선순위

### MVP (1차 출시 — 지금 만든 버전에 포함)
우선순위 P0는 "이게 없으면 제품이 아님" 수준입니다.

| 우선순위 | 기능 | 설명 |
|---------|------|------|
| **P0** | 거래 입력 | 개인/사업 × 수입/지출, 금액·카테고리·날짜·메모 |
| **P0** | 개인/사업 분리 | 한 번의 토글로 두 영역을 나눠 기록 |
| **P0** | 월별 대시보드 | 사업 손익 / 개인 수지 / 전체 순흐름 한눈에 |
| **P0** | 데이터 저장(로컬) | 새로고침해도 데이터 유지 (localStorage) |
| **P1** | 카테고리 분류 | 세금 신고에 도움되는 비용 항목 체계 |
| **P1** | 최근 6개월 추세 | 수입/지출 막대 그래프 |
| **P1** | CSV 내보내기 | 세무사 전달·엑셀 가공용 |

### 2차 (Next)
- 정기 거래 자동 등록 (월 임대료·구독료 등)
- 간이 부가세/예상 세액 가늠 위젯
- 자산(현금·통장·카드) 잔액 관리, 순자산 추이
- 클라우드 동기화(여러 기기에서 같은 데이터) → 백엔드 필요
- 영수증 사진 첨부

### 3차 (Later)
- 거래처/고객 관리(미니 CRM)
- 간이 세금계산서·견적서 발행
- 은행/카드 내역 자동 연동(오픈뱅킹)

## 3. 차별화 포인트 (USP)

1. **"한 사람의 두 지갑" 모델** — 프리랜서·1인 사업가는 사업비와 생활비가 뒤섞입니다. 결은 *분리 입력 + 통합 조회*로 "이번 달 진짜 내가 남긴 돈"을 보여줍니다.
2. **세금 친화적 카테고리 설계** — 비용 항목이 처음부터 임차료·인건비·지급수수료 등 신고 시 쓰이는 분류와 맞물려 있어, 연말·반기 정산 시 CSV 한 장으로 끝.
3. **무설치·즉시 실행** — 앱스토어 등록·회원가입·서버 없이 브라우저로 바로 사용. 데이터는 내 기기에만 저장돼 프라이버시 부담이 적음.
4. **모바일 우선 입력 UX** — 하단 탭바·큰 금액 입력칸·세그먼트 버튼으로 길에서도 3초 입력.

> 한 문장 USP: **"가계부처럼 쉽게 적는데, 사업 손익까지 정리되는 1인 사업가용 머니 매니저."**

---

## 4. 타깃 유저 & 핵심 시나리오

- **지식창업자/프리랜서**: 용역비가 들어오면 사업 수입으로, 노트북 구독료는 사업 비용으로, 점심은 개인 지출로. 월말에 "사업 손익"만 따로 확인.
- **소상공인**: 매출·매입·고정비를 빠르게 기록하고 카테고리별 비용을 점검.
- **분리 관리를 원하는 개인**: 부업 수입과 본업/생활비를 섞지 않고 보고 싶은 사람.

## 5. 데이터 모델 (DB 설계 기초 아이디어)

현재 MVP는 브라우저 `localStorage`에 JSON 배열로 저장합니다. 추후 백엔드(예: SQLite/PostgreSQL, 또는 Firebase)로 확장할 때 아래 스키마를 그대로 옮기면 됩니다.

### 5-1. `transactions` (거래) — MVP의 단일 핵심 테이블
| 필드 | 타입 | 설명 |
|------|------|------|
| id | string(PK) | 고유 ID |
| date | date | 거래일 (YYYY-MM-DD) |
| scope | enum | `business` / `personal` |
| type | enum | `income` / `expense` |
| category | string | 카테고리명 (FK 후보) |
| amount | integer | 금액(원, 양수) |
| memo | string | 메모 (선택) |
| created_at | datetime | 생성 시각 |

```json
{
  "id": "1730000000-ab12c",
  "date": "2026-06-04",
  "scope": "business",
  "type": "income",
  "category": "용역/수수료 수입",
  "amount": 1500000,
  "memo": "클라이언트 A 6월 용역비"
}
```

### 5-2. 확장 시 추가될 테이블 (2차 이후)
- **`categories`**: `id, scope, type, name, tax_code(세금 분류 코드), sort_order`
- **`accounts`** (자산): `id, name, kind(cash/bank/card), balance` → 순자산 계산
- **`recurring`** (정기거래): `id, transaction_template, interval(monthly 등), next_date`
- **`users`** (클라우드 동기화 시): `id, email, ...` + 위 테이블에 `user_id` FK 추가

### 5-3. 핵심 계산 로직
```
사업 손익  = Σ(business·income) − Σ(business·expense)
개인 수지  = Σ(personal·income) − Σ(personal·expense)
순흐름(전체) = (모든 income) − (모든 expense)
```
모두 "선택한 달" 기준으로 필터링하여 집계합니다.

| 단계 | 프론트 | 데이터 | 비고 |
|------|--------|--------|------|
| **지금 (MVP)** | HTML+CSS+Vanilla JS | localStorage | 무설치, 단일 기기 |

## 8. 수익 모델 (Monetization Model)

### 8-1. 모델 비교

| 모델 | 적합도 | 비고 |
|------|--------|------|
| **Freemium 구독(SaaS)** | ◎ **추천** | 1인 사업가는 세금·손익 관리에 월 정액 지불 의향 ↑ |
| 세무사/전문가 연계 수수료 | ○ | B2B2C, 신고철 한정 수익. 2차 레버리지 |
| 광고 | △ | 사업가 타깃엔 객단가 낮고 신뢰 훼손 |
| 단건 결제(리포트/내보내기) | △ | 보조 수익. 구독 거부층 흡수용 |
| 팀/다중 사용자(B2B) | ○ | 소상공인 직원 공유, 객단가 최상위 |

핵심 전략은 **Freemium 구독**입니다. 그리고 **웹/PWA를 유지하는 것이 결정적 이점** — 앱스토어 인앱결제 수수료(15~30%)를 우회해 PG 수수료(2~3%)만 부담하면 됩니다.

### 8-2. 플랜 설계 (예시)

| 플랜 | 가격(안) | 핵심 |
|------|---------|------|
| **Free** | ₩0 | 개인 가계부 전 기능 + 사업 기록 *제한*(월 거래 N건·자산 N개·최근 3개월 조회) |
| **Pro** | ₩4,900/월 · ₩49,000/년 | 무제한 + 사업 손익/세금/리포트/통계 고급 + OCR + 다기기 |
| **Business** | ₩19,900/월 | Pro + 거래처/미수금 관리 + 견적/세금계산서 + 멤버 공유 |

**무엇을 Pro 뒤로 가둘 것인가 (지불 의향이 높은 순):**
1. 세금 관련 전부 — 부가세·예상세액·세금 캘린더·세무 리포트 (*돈을 아껴주는 직접 가치*)
2. 무제한 — 거래·자산·정기거래·조회 기간
3. 고급 통계 — 연간 비교, 예산 대비, 카테고리 추이
4. 내보내기 — PDF 손익계산서·엑셀·세무사 패키지
5. 영수증 OCR 자동 입력 (입력 마찰↓ → 리텐션↑)

> Free는 "개인 가계부"로 충분히 쓸 만하되, **사업가가 한손을 진지하게 쓰는 순간 자연스럽게 Pro가 필요**해지도록 경계를 긋는 것이 핵심입니다.

## 10. 수익화를 위한 아키텍처 전환

지금 구조(거대 단일 `app.js` + Firestore 직접 접근)는 베타엔 충분하지만, **돈과 권한이 들어오면 보안·신뢰성 요구가 급상승**합니다.

| 영역 | 현재(As-Is) | 수익화 단계(To-Be) |
|------|------------|--------------------|
| 권한/결제 | 없음 | Cloud Functions 서버 검증 + Firestore 보안 규칙 강화 (plan 위변조 방지) |
| 코드 구조 | 단일 `app.js` | 즉시 React 전환보다 **게이팅·결제 모듈만 분리**. 규모 커지면 컴포넌트화 |
| 데이터 신뢰 | 로컬+클라우드 동기화 | 돈 다루는 앱 → **백업/복구·감사로그·충돌 해결** 강화 (유료층은 데이터 손실에 민감) |
| 분석 | 없음 | **GA4/Amplitude로 전환 퍼널**(가입→활성→체험→결제→유지) 측정. 어디서 결제 벽을 만나는지 추적 |
| 온보딩 | 즉시 입력 | 첫 경험 가이드 + Pro 체험 유도 |

> 보안 원칙 한 줄: **구독 상태는 절대 클라이언트만 믿지 않는다.** 결제 검증·entitlement 기록은 서버에서, 프론트는 그 결과를 *읽기만* 한다.

## 11. 성공 지표 (North-Star & KPI)

- **North-Star:** 유료 활성 사업가 수 (Paying Active Sellers)
- **수익:** MRR, ARPU, LTV
- **퍼널:** 가입→활성(첫 거래 입력)→체험 시작→유료 전환율→리텐션(D30)→Churn
- **벤치마크 가설:** 무료→유료 전환 2~5%, 월 churn 5% 이하면 건강
- **선행지표:** 주간 거래 입력 빈도, 세금/리포트 기능 사용률(전환 직결)

## 12. 단계별 실행 로드맵 (요약)

```
지금 ──────────────────────────────────────────────▶ 매출
  │
  ├─ Phase 1 (결제 토대)      : 구독 인프라 + Entitlement + 게이팅 + 14일 체험 + Paywall
  │     └ 목표: "결제가 되는" 상태. 첫 유료 사용자 확보
  │
  ├─ Phase 2 (가치 기능)      : 예산·알림 / 세금 캘린더·예상세액 / OCR / PDF 리포트
  │     └ 목표: Free→Pro 전환율·리텐션 끌어올리기
  │
  └─ Phase 3 (확장)          : 거래처 CRM·미수금 / 견적·세금계산서 / (후순위) 오픈뱅킹 / 팀
        └ 목표: 객단가↑ Business 플랜, B2B 확장
```

### 리스크 메모
- **국내 PG 정기결제 심사**: 사업자등록·계약 필요. 일정에 선반영.
- **개인정보·전자금융 규제**: 오픈뱅킹/세금계산서는 규제 영역 → 법무 검토 후 진입.
- **앱스토어 vs PWA**: 인앱결제 수수료 회피를 위해 **웹 결제 PWA를 메인으로 유지**. 네이티브는 알림·설치 경험이 필요할 때만 래핑(WebView) 고려.
- **세금 기능의 책임 한계**: "참고용 추정"임을 명시해 세무 책임 리스크 차단.
